Where and how BM25 is used

Overview for the legal-document-assistant project. BM25 is not stored in Qdrant; it runs in Python (rank_bm25.BM25Okapi) only on the text of chunks already returned by dense retrieval.

Quick legend

  • Qdrant — dense vectors (cosine), persistence
  • BM25 — lexical, in-memory, over the candidate pool
Single BM25 touchpoint in code: HybridRetriever.retrieve() — after vector_store.search(), before the final sort. File: src/legal_ai/retrieval/hybrid_retriever.py.

API chain: dependencies.qa_chain → QAChain.answer → HybridRetriever.retrieve. Ingestion, RiskDetector, and the document comparator do not use BM25.

Flow inside HybridRetriever.retrieve

candidate_k = max(top_k * 4, 20) — BM25 only sees these candidates, not the full index. 

flowchart TD
  start([Query text])
  start --> empty{Query empty?}
  empty -->|yes| z([Return empty list])
  empty -->|no| k[Compute top_k and candidate_k]
  k --> enc[Embedder.encode_query]
  enc --> qsearch[QdrantVectorStore.search dense vector]
  qsearch --> nohits{Zero hits?}
  nohits -->|yes| z
  nohits -->|no| tok[Regex tokenization: chunk text + query lowercased]
  tok --> bm25[BM25Okapi over token lists from candidates]
  bm25 --> scores[bm25.get_scores query_tokens]
  scores --> n1[Min-max Qdrant scores]
  scores --> n2[Min-max BM25 scores]
  n1 --> fuse["Fused score w_dense * norm_dense + w_bm25 * norm_bm25"]
  n2 --> fuse
  fuse --> sort[Sort descending by fused_score]
  sort --> cut[Take first top_k as RetrievedChunk]
  cut --> endNode([Back to QAChain])

What data BM25 sees

  • Input: the same question string used for embedding; the list of SearchHit objects from Qdrant (each chunk’s text field).
  • Output: a vector of raw scores; then min-max normalization and combination with the normalized dense score (retrieval_dense_weight, retrieval_bm25_weight in Settings).
